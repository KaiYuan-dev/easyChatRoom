# 架构决策记录（ADR）

> **ADR = Architecture Decision Record**
> 每一份 ADR 记录一次有意义的架构或技术决策的**完整上下文**。
> 维护者：**CTO**（由 `/CLAUDE.md` 角色负责）。

---

## 为什么存在

代码只告诉你"是什么"。ADR 告诉你**"为什么"**——当时考虑了什么、否决了什么、为什么选了这个。

没有 ADR 的项目，几个月后：

- 新加入的人不敢改任何设计，因为不知道原决策是否仍成立
- 老成员也忘了当初的权衡，反复踩同一个坑
- 想改进某处架构的人，花时间重走了一遍决策过程

ADR 让决策的**推理过程**和结论本身一样被持久化。

---

## 核心规则

### 1. 不可变（Immutable）

**一旦 ADR 被接受（Accepted）就不再修改正文**。

- 如果决策本身要变更，**写新的 ADR** 并把旧 ADR 的状态改为 `Superseded by ADR-XXXX`
- 只修改正文中的明显事实错误（错别字、链接失效等），不修改决策推理
- 这样保留完整历史，不允许事后美化

### 2. 小而多，一决策一文件

不要写成一本册子。每个决策独立文件，按顺序编号。

### 3. 面向未来的读者

阅读者可能是 3 个月后的你、6 个月后的新人、甚至 2 年后的自己。不要依赖"当时大家都懂"的隐含信息，把背景写清楚。

### 4. 有状态

每份 ADR 标明当前状态（见下文）。

---

## 文件命名

```
ADR-XXXX-kebab-case-title.md
```

- `XXXX`：4 位数字，按全局顺序递增（`0001` `0002` ... `0047`）
- 标题用小写 kebab-case，简洁描述决策
- 示例：`ADR-0001-modular-monolith.md`、`ADR-0007-ack-as-durable.md`

**不复用已废弃的编号**。即使某 ADR 状态变为 Superseded 或 Rejected，编号保留。

## 状态枚举

| 状态 | 含义 |
|---|---|
| `Proposed` | 已提议，尚未采纳（讨论中） |
| `Accepted` | 已采纳，当前有效 |
| `Deprecated` | 不再推荐但未被替代（例如功能废弃后附带的决策） |
| `Superseded by ADR-XXXX` | 已被新决策取代，正文保留为历史 |
| `Rejected` | 提议后被否决（仍保留文件作为"考虑过但没选"的记录） |

## 何时创建 ADR（触发条件）

CTO 必须在以下情况新建 ADR：

- **技术选型**：选数据库、MQ、框架、通信协议等
- **架构模式**：单体 vs 微服务、分层方式、模块划分原则
- **协议/契约**：对外 API 风格、ACK 语义、序列化格式
- **安全/合规**：鉴权方式、数据加密、审计策略
- **不显而易见的业务决策**：例如"不对客户端提供历史消息"这种有讨论空间的业务约束
- **否决一个看起来合理的方案**：即使最终选 A 没选 B，B 的否决也值得记录

**不需要 ADR 的场景：**

- 常规 CRUD 实现细节
- 小工具类、常量值、日志格式等实现层选择
- 显然正确、无争议的选择（例如"代码要 UTF-8 编码"）

判定标准：**这个决策在未来有没有人可能想改？改之前要不要理解原理由？** 是 → 写 ADR。

## 工作流

### 新决策

1. 用户与 CTO 讨论并敲定决策
2. CTO 复制 `TEMPLATE.md` 为新文件 `ADR-XXXX-title.md`
3. 填写正文，状态设为 `Accepted`（或 `Proposed` 若仍在讨论）
4. 在对应 `CLAUDE.md` 的 ADR 速查表（§11 或等价位置）追加一行，引用 ADR 编号
5. commit 时包含 ADR 文件

### 决策变更

1. 新建 `ADR-YYYY-new-title.md`，正文说明为什么推翻或调整
2. 在新 ADR 的 "Relations" 部分填 `Supersedes ADR-XXXX`
3. **打开旧 ADR**，**仅修改状态栏**为 `Superseded by ADR-YYYY`（正文其他部分不改）
4. 更新速查表

### 发现决策行不通但还没替代方案

- 状态改为 `Deprecated`，备注"待定替代方案"
- 在 `FEATURES.md` 待决策表新增对应条目

## 与其他文档的关系

| 位置 | 职责 |
|---|---|
| `docs/ADR/ADR-XXXX-*.md` | **完整决策记录**（详细），不可变 |
| `backend/CLAUDE.md` §11 | 后端相关决策**速查表**（一行一条），可更新状态引用 |
| `frontend/CLAUDE.md` §N | 前端相关决策速查表（如有） |
| `/CLAUDE.md`（CTO） | 不维护速查表，但在裁决跨团队冲突时必须**创建或引用 ADR** |

**双向引用**：ADR 文件正文引用相关 `CLAUDE.md` / `FEATURES.md` / `API_SPEC.md` 的具体章节；速查表与文档引用 ADR 编号。
