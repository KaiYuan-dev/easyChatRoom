# ADR-0007: ACK 语义定义为"已持久化不丢"（单次 ACK）

- **状态**: Accepted
- **日期**: 2026-04-16
- **决策者**: CTO（代表全队）
- **相关**:
  - Relates to: `backend/CLAUDE.md` §6.5 / `docs/API_SPEC.md` §3.5.1
  - Relates to: FEATURES F-047

---

## 背景

设计聊天室消息的客户端 ACK 协议时，需要明确 `ACK` 消息对客户端意味着什么——这直接影响：

- 客户端消息 UI 的状态机（何时显示"已送达"）
- 客户端的重发逻辑（什么情况下应该重试）
- 服务端的责任边界（服务端承诺了什么）
- 代码实现（Gateway 与 Message 模块的协作方式）

业界常见 IM 的 ACK 设计有两类：

1. **多段 ACK**：RECEIVED（服务端收到网络包）→ ACCEPTED（已入库）→ DELIVERED（已送达对端），每段一次回包
2. **单次 ACK**：只一次回包，但需定义清楚"回的时候服务端做到了什么"

初稿协议设计时采用了多段 ACK（RECEIVED / ACCEPTED / FAILED 三种）。但在与用户讨论时，出现了关于 ACK 语义的根本分歧。

---

## 考虑过的方案

### 方案 A：三段 ACK（RECEIVED / ACCEPTED / FAILED）

Gateway 收到客户端消息 → 立即回 `RECEIVED`（服务端已收到网络包） → 调 Message 模块持久化并广播 → 完成后回 `ACCEPTED` 或 `FAILED`。

**优点：**
- 客户端反馈快（`RECEIVED` 几乎零延迟）
- 细分状态便于观察链路

**缺点：**
- `RECEIVED` 的语义是"服务端已收到网络包"，**但服务端此时没有做任何持久化**——如果此刻服务端崩溃，消息会丢失
- 客户端 UI 状态机复杂（三段 + 失败 = 四态）
- 协议歧义容易导致误理解：用户可能以为 `RECEIVED` 就代表消息安全了

### 方案 B：单次 ACK，语义 = "已入库"

Gateway 收到消息 → 生成 serverMsgId → **同步**调 Message 模块持久化 → 成功后回一次 `ACK(status=OK)`；失败回 `ACK(status=FAILED)`。广播 Redis Pub/Sub **异步**进行。

**优点：**
- ACK 语义清晰：ACK 即"服务端承诺不丢"
- 客户端 UI 简单（二态：OK / FAILED + 等待中）
- 与 Kafka / RabbitMQ 等消息中间件的 ACK 行业惯例一致
- 广播失败不影响 ACK 语义（消息已在 DB，接受实时到达损失——本项目反正不提供历史）

**缺点：**
- `RECEIVED` 的"立即反馈"价值失去——客户端必须等持久化完成才收到 ACK
- 客户端无法区分"网络已达但服务端处理中"与"网络未达"

### 方案 C：单次 ACK，语义 = "已广播"

Gateway 持久化 + 广播都完成后才回 ACK。

**优点：**
- 语义更强

**缺点：**
- 等待时间更长（Redis Pub/Sub 虽快但 ACK 路径拉长）
- 广播失败要影响 ACK 语义，反而让 ACK 承担了它不该承担的责任
- 聊天室场景广播是 best-effort（成员众多），不适合做成 ACK 门槛

---

## 决策

**采用方案 B：单次 ACK，语义为"消息已持久化，服务端承诺不丢"**。

具体实现：

- Gateway 收到 `MSG` 后**生成 serverMsgId**（雪花），传给 Message 模块
- Message 模块**同步**完成校验 + MongoDB 持久化，成功后返回
- Gateway 回 `{type:"ACK", clientMsgId, serverMsgId, ts, status:"OK"}`
- ACK 完成后，Gateway **异步**触发 `MessageService.broadcast(...)`，后者 Redis PUBLISH
- 校验失败（长度、限流、成员等）回 `{type:"ACK", status:"FAILED", code, msg}`
- 广播失败仅记 error 日志，不反馈客户端

---

## 理由

ACK 协议的**核心价值**是给客户端一个**可信赖的"服务端已接管"信号**。方案 A 的 `RECEIVED` 语义（网络已收到）对客户端几乎没有价值——客户端自己的 TCP 栈都知道对方收到了。真正有价值的信号是"服务端已把消息写入持久存储，承诺不丢"。

方案 C 把广播成功绑定到 ACK，引入的问题多于解决的：广播本身是 best-effort，失败率可能比持久化高，让客户端 UI 对广播成功进行反应会增加 UI 复杂度和错误提示噪音。

MongoDB 同机房写入延迟通常 < 10ms，方案 B 的"等持久化"延迟在用户体感上可以忽略——这个延迟代价换来的是**极其清晰的语义**。

此外，选 B 对应业务规则的另一个决策：**不对客户端提供历史消息**。这意味着一旦消息持久化（即便没广播成功）服务端也不需要补发——消息"就这样算完"。这与 B 的语义天然吻合。

---

## 后果

### 积极后果

- 客户端 UI 状态机简单（三态：发送中 / 已发送 / 失败）
- 协议语义清晰，难以误解
- 未来扩展私聊 / 群聊时，可在此基础上**增加** `DELIVERED` 段（per-receiver），不影响 CHATROOM 的现有 ACK 行为
- 与工业界消息中间件惯例一致，开发者认知迁移成本低

### 消极后果 / 代价

- 客户端需要等待持久化完成才得到反馈——虽然通常 < 10ms，但在 DB 响应异常时客户端会感觉"一直在转圈"
- 广播失败的场景下，其他客户端收不到消息，但发送者已认为成功——这种**不一致**需要业务接受
- 失去 `RECEIVED` 这种可用于监控"网络 vs 处理"分层的信号——监控需要通过其他方式实现

### 未来需要关注的风险

- 若未来决定提供历史消息，可能需要重新审视 ACK 语义——此时 ACK 可以仍是"已持久化"，但客户端可能期待额外的"对端已读"等信号。这会触发新 ADR，不影响当前决策
- 若 MongoDB 持久化成为 ACK 路径的瓶颈（例如写入延迟飙升），需要考虑异步化或 Write-Ahead Log 机制。届时可能出新 ADR 调整持久化层
- `im-core` 抽象中已为 PRIVATE / GROUP 类型预留了多段 ACK 扩展位（`DELIVERED`），聊天室类型按本决策仅用单段

---

## 实施要点

- `serverMsgId` 由 Gateway 生成（见 ADR-0011），传入 Message 模块复用
- `MessageService` 的对外接口拆成 `persist`（同步）和 `broadcast`（异步）两个方法
- 广播失败只记 error 日志，**不**重试 ACK——已 ACK 的消息不允许再回翻转状态
- 客户端 ACK 等待超时定为合理值（如 10s），超时 UI 显示"失败，可重试"
