# ADR-0001: 采用 Modular Monolith 架构

- **状态**: Accepted
- **日期**: 2026-04-16
- **决策者**: CTO（代表全队）
- **相关**:
  - Relates to: `backend/CLAUDE.md` §1 / §3 / §4

---

## 背景

启动"附近聊天室"后端项目时，面临架构粒度的初始选择：

- **业务特点**：LBS（位置聚合）+ 实时聊天，模块职责清晰（用户、鉴权、位置、聊天室、消息、网关）
- **预期规模**：前期并发数百到数千，未来可能扩展到数万
- **团队规模**：当前小团队，不具备维护多服务 + 完整 DevOps 的人力
- **核心矛盾**：既希望现在能快速开发（少运维、少跨服务调试），又不希望未来要做微服务化时被迫大规模重构

这个选择必须在 Phase 1 开始前敲定，影响模块依赖方向、事务边界、状态存储位置等所有后续决策。

---

## 考虑过的方案

### 方案 A：直接上微服务

把每个业务域（用户、鉴权、位置、聊天室、消息、网关）从第一天就拆成独立服务。

**优点：**
- 未来规模化无需重构
- 服务间天然隔离

**缺点：**
- 小团队运维负担重（N 份部署、N 份监控、服务注册发现、分布式追踪）
- 跨服务调用调试复杂
- 前期需求尚不稳定，服务边界可能错，返工成本高
- 过度设计——当前规模完全不需要

### 方案 B：传统单体（无模块边界纪律）

单一项目，按技术分层（Controller / Service / Dao）组织，不强制领域边界。

**优点：**
- 最简单、最快
- 本地事务无压力

**缺点：**
- 代码会随时间"泥球化"，所有东西都可以调所有东西
- 想拆微服务时几乎等于重写
- 跨领域耦合无法识别

### 方案 C：Modular Monolith（模块化单体）

单一部署单元，但代码按业务域分模块，模块之间只通过 `api` 包通信，**按分布式前提编写代码**（禁止本地锁保护共享状态、禁止跨模块事务等）。

**优点：**
- 运维简单（一份部署）
- 架构纪律强，模块可独立拆分
- 未来拆微服务只需把本地 Service 调用换成 RPC
- 有助于早期暴露真实模块边界

**缺点：**
- 需要持续维护模块边界纪律（靠 ArchUnit + Review）
- 写代码时必须克制"随手调一下别的模块"的便利
- 团队成员需要理解这套约束

---

## 决策

**采用方案 C：Modular Monolith**。

具体落地：

- 按业务域划分 10 个 Maven 模块（`common`、`infrastructure`、`user`、`auth`、`location`、`chatroom`、`im-core`、`message`、`gateway`、`app`）
- 模块间只允许引用 `api` 包；`internal` 包跨模块 import 禁止
- 代码必须按**多实例部署前提**编写，即使当前是单实例
- Phase 7 用 ArchUnit 测试强制约束

---

## 理由

方案 A 对当前规模是明显的过度设计，运维负担 > 架构收益。

方案 B 的最大风险是**不可逆性**：一旦代码泥球化，再想拆就要大规模重写。而项目明确预见未来要规模化。

方案 C 以"少量纪律成本"换取"拆分可选项"，在当前场景是最佳权衡。纪律成本主要通过两点控制：

1. **显式约束写入 `backend/CLAUDE.md` §3**（架构铁律）——不依赖个人自觉
2. **ArchUnit 自动化拦截**——Phase 7 上线，之后违规即构建失败

---

## 后果

### 积极后果

- 起步速度接近传统单体，但代码具备未来拆分能力
- 模块边界被强制显式化，早期能暴露设计问题
- 运维简单，小团队可持续
- Review 时有明确的架构规则可检查

### 消极后果 / 代价

- 开发者需要忍受"看似冗余"的 api 包定义（本地调用也要过接口）
- 新成员需要学习约束，有理解成本
- 某些情况下无法使用本地事务跨模块，必须走事件 + 最终一致性，比跨模块直接 `@Transactional` 更复杂

### 未来需要关注的风险

- 如果团队纪律松懈，单体会退化为泥球。**关键防线是 ArchUnit**，Phase 7 必须落实，不能推延
- 模块划分若初期错了，调整边界成本仍然不低（不如微服务那么僵化，但也不是免费）
- 某个模块的 `api` 包会随时间增长，若超过合理规模说明该拆出去

---

## 实施要点

1. Phase 1 必须搭好 Maven 多模块骨架与依赖方向
2. 每个模块统一内部结构（`api` / `internal` / `web` / `config`）
3. `internal` 包下的类**尽量 `package-private`**，让编译器挡跨模块引用
4. Phase 7 用 ArchUnit 写至少三类测试：
   - 模块依赖方向正确
   - `internal` 包不被外部 import
   - 禁止 `synchronized` 保护跨请求状态
