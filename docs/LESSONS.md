# Lessons Learned · 经验教训库

> **维护者**：CTO（`/CLAUDE.md` 角色）
> **读者**：CTO（本人）+ 在派发任务时引用给 sub-agent
> **作用**：把重复性痛点、踩过的坑、根本原因沉淀为可检索的知识，对抗 AI 无记忆特性
> **协议**：详见 `/CLAUDE.md` §11.6 学习机制

---

## 为什么要有这份文档

Claude session 会失忆。沉默的经验随 session 终结而消失。这份文档把经验**结构化沉淀到文件系统**，让下一个 session 的 CTO 能：

1. **启动时感知历史痛点**（`/CLAUDE.md §1` 第 6 步扫描近 7 天新增）
2. **派发任务前引用相关经验**（任务简报 `## 相关经验（若有）` 字段）
3. **Review 时借助历史教训识别重复性问题**

这是用文档实现的"机构记忆"（institutional memory）。

---

## 创建条件（触发一条 Lesson）

以下事件发生时，CTO **必须**在本文件追加一条：

| 触发 | 来源 |
|---|---|
| 同一 bug 3 次 QA 失败升级给用户 | `/CLAUDE.md §6.7.2` |
| Sub-agent 重复违反同一条架构铁律（≥3 次） | CTO Review 时发现 |
| 文档内部发现不一致并修复（如并行 session 造成的冲突） | 文档 audit |
| 重大架构决策后补充"教训"维度（ADR 无法覆盖的） | ADR 定稿时 |
| 用户明确说"这条要记下" | 用户指令 |
| 集成测试暴露跨层系统性问题 | QA 报告 |

**不值得记录**：一次性的小错、明确的新手 mistake、已有 ADR 覆盖的权衡。避免与 ADR 重复。

**涉及**：T-XXX / F-XXX / ADR-XXXX / 文档章节（若相关）

#### 症状（发生了什么）
<具体现象，越细越好。未来的你读这段应该能"看到当时发生了什么"。>

#### 根因
<为什么会这样？不要停在表面原因（"忘记了"），挖到机制层（"因为没有 CI 拦截"）。>

#### 教训
- <从此以后应该怎么做：规则/原则 1>
- <规则/原则 2>

#### 反哺机制（关键 · 必填）
<光写在这里没用，必须落入某个可执行位置。至少勾选一项。>

- [ ] 加入了 `code-review/CLAUDE.md` 检查清单
- [ ] 加入了 ArchUnit 测试
- [ ] 仅存此处（下次同类任务派发时在简报中手动引用 L-XXXX）

## 反哺机制说明

一条 Lesson 的价值取决于它被"用上"的次数。常见反哺路径：

1. **规则类教训** → 写入对应 `CLAUDE.md` 的铁律或规范章节
2. **可检测的代码模式** → 加入 `code-reviewer` 检查清单或 ArchUnit 测试
3. **流程类教训** → 更新 `/CLAUDE.md` 工作流或 Review 清单
4. **任务经验**（"这类任务容易在哪翻车"）→ 仅存此处；CTO 派发同类任务时在简报 `## 相关经验` 字段手动引用 `L-XXXX`

反哺是否落实，在 `/CLAUDE.md §14` 自检清单中检查。

---

## 衰减与归档规则

- **近 90 天**的条目保留在本文件顶部（"近期 Lessons"区）
- 超过 90 天 **且**已反哺到其他文档的条目 → 归档至 `docs/lessons-archive/YYYY-QN.md`
- **永不删除**。即使教训已过时，保留历史痕迹用于审计
